Django项目中采用celery异步任务执行+redis任务队列数据存储+执行任务结束回调过程。

前端发送sync请求，执行任务首先判断conf表中的action_ops状态字段是否为running, running就退出。不是running，改成running，操作表中增加一条数据({“ops_2_conf”: 17, “operator”: “super admin”, “action_name”: “sync”})，拿到这条数据id;然后操作详情表中写数据(记录执行的动作具体是什么);然后先从数据库中查询配置，写入本地，写入成功，更新动作表状态。然后执行耗时的异步操作，然后判断nginx类型，如果是consul类型(加一步动作add_dump)，然后调用ansible runner执行动作，先执行add_dump动作， 查询到执行命令，交给ansible异步去执行，异步那里拿到ansible执行命令的状态和结果(状态码0，执行结果)，然后将操作详情记录到动作详情表；然后执行sync动作，调用subprocess将配置文件传送至远程管理主机，再调用ansible执行远程主机上的脚本，然后拿到执行后的返回值，更新操作详情表成功。然后返回状态，自动存储至redis。

执行成功，返回前端任务id；前端可以定时向后端发请求，带着任务id。

ansible resultcallback 插件用来实现对 ansible 执行结果的精确调用， callback 是 ansible 的任务结果处理插件，也将作为整个平台的任务反馈机制，提供任务执行信息。我们在 callback 中初始化产生一个任务 ID 作为 redis key，redis value 将使用 list 类型
