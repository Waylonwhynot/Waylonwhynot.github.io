服务网格、Envoy和istio入门

1. Kubernetes 与 Service Mesh是什么关系？

一个分布式应用的构建满足4个需求：网络管理，状态管理，绑定管理，生命周期管理

kubernetes解决了哪些问题？

  • Lifecycle：通过容器编排系统，解决了应用周期Lifecycle 生命周期管理；kubernetes只是分布式应用程序当中的生命周期管理工具
    • packaging 镜像
    • healthcheck
    • deployment
    • Scaling
    • Configuration-通过configmap解决
  • Networking
    • 借助于CNI网络插件解决了基本的网络通信
    • 服务注册和服务发现可以借助于Service组件来理解这个概念；但service组件本身虽然有服务发现和注册的功能，但这种功能过于底层和通用，很难满足应用程序自身的真正需要；借助于coredns通过服务名称来完成自动发现，超时和重试，做不到这一点；应用程序自己要解决这个问题
    • 所以网络恰恰就是服务网格service mesh要解决的问题；也就意味着在k8s的基础之上，再部署一层istio这样的服务网格组件，才能真正的为应用程序(分布式应用)再提供一层保障
  • Binding(kubernetes解决不了，istio也没这个能力)
    • serverless 一定程度上解决了binding，部分有实现
    • Dapr 解决
    • 概念理解（绑定角度面临的问题，这些都要通过大量的内部规范，内部契约来解决问题）
      • connector：连接器，至少被访问端要有api接口；api接口与调用端和被调用端之间要有接口契约，必须要基于某种规范进行开发
      • protocol conversion：协议转换
      • message transformation：消息格式转换传输 json xml
      • message routing：消息路由
      • transactionality：事务
  • State（依然解决不了）
    • serverless 一定程度上解决了state，部分有实现
    • Dapr解决

所以真正意义上构建出一个完整的云原生的应用，k8s下一个阶段就是在其基础上部署一个服务网格系统，再部署一个serverless，而Dapr一定是下一个争夺的热点
